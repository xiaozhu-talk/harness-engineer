# 评估标准

> 本文件定义 Evaluator 使用的评判标准。权重和阈值可根据项目类型调整。

## 评分维度

### 1. 产品深度 (Product Depth)
**权重**: ★★★ | **阈值**: ≥ 7/10

评估功能是否有深度，还是只实现了表面。

| 分数 | 描述 |
|------|------|
| 9-10 | 功能完整且有深度，考虑了边界情况和高级用例 |
| 7-8 | 核心功能完整，有一定深度 |
| 5-6 | 基本功能可用，但缺少应有的深度 |
| 3-4 | 只有骨架实现，很多功能是 stub |
| 1-2 | 几乎没有可用的功能 |

---

### 2. 功能性 (Functionality)
**权重**: ★★★ | **阈值**: ≥ 8/10

评估用户能否完成核心任务，是否有阻塞性 Bug。

| 分数 | 描述 |
|------|------|
| 9-10 | 所有核心和次要功能正常工作，无 Bug |
| 7-8 | 核心功能正常，有少量 Minor Bug |
| 5-6 | 核心功能基本可用，有间歇性 Bug |
| 3-4 | 核心功能部分损坏，无法完成主要任务 |
| 1-2 | 应用无法正常使用 |

---

### 3. 设计质量 (Design Quality)
**权重**: ★★ | **阈值**: ≥ 6/10

评估视觉设计是否协调、有整体感。

| 分数 | 描述 |
|------|------|
| 9-10 | 博物馆级设计——独特、大胆、有完整的设计语言 |
| 7-8 | 专业级——配色、排版、布局形成整体 |
| 5-6 | 合格但平庸——能用但没有特色 |
| 3-4 | 明显的默认模板风格 |
| 1-2 | 视觉混乱，没有设计可言 |

**反面模式检测**（如果发现以下模式，设计质量自动 -2）：
- 紫色/蓝色渐变 + 白色卡片（AI 默认风格）
- 未修改的 Material/Ant Design 默认组件
- 过度使用阴影和圆角
- 配色不协调（超过 5 种主色调）

---

### 4. 原创性 (Originality)
**权重**: ★★ | **阈值**: ≥ 6/10

评估是否有自定义设计决策，还是完全靠模板。

| 分数 | 描述 |
|------|------|
| 9-10 | 有明显的创意决策——人类设计师会认可 |
| 7-8 | 有自定义设计选择，不是千篇一律 |
| 5-6 | 在模板基础上有些许定制 |
| 3-4 | 基本是默认模板 |
| 1-2 | 完全的 AI 默认生成风格 |

---

### 5. 代码质量 (Code Quality)
**权重**: ★ | **阈值**: ≥ 7/10

评估代码是否可维护、结构清晰。

| 分数 | 描述 |
|------|------|
| 9-10 | 代码清晰、文档完善、架构优雅 |
| 7-8 | 代码结构合理、命名清晰、有适当注释 |
| 5-6 | 代码可读但结构松散 |
| 3-4 | 代码混乱、重复多、方法过长 |
| 1-2 | 代码不可维护 |

---

## 评分规则

1. **独立评分**：每个维度独立评分，不受其他维度影响
2. **硬阈值**：任何一个维度低于阈值即驳回整个 Sprint
3. **具体化**：评分必须附带具体的证据和说明
4. **严格偏向**：当犹豫时，选较低分——宁可严格也不宽松
5. **重复不扣分**：同一问题出现多次只算一次扣分
